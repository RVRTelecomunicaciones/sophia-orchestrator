# Delta: external-mcp-proxy

> **Lifecycle reinterpretation (design D-M4-2, authoritative)**: the proposal's
> "spawned_per_change with close-on-change-end" is NOT implementable in
> agent-mcp — proxied tool calls arrive as independent stateless MCP calls with
> no change_id (`agentrun/request.go:43`) and nothing signals change end. The
> implemented lifecycle is **persistent-per-process**: lazy spawn on first tool
> call, session reused for the process lifetime, reaped on `App.Stop`. This is
> correct because `graphify serve` hot-reloads `graph.json` via mtime, so a
> long-lived sidecar never serves stale data. The `spawned_per_change` config
> label is retained as documented intent only.

## Capability

A lazy-spawn sidecar proxy: spawns one MCP client adapter on the first proxied
tool call for a provider, caches the session for all subsequent calls during
the process lifetime, and closes the subprocess on application shutdown
(`App.Stop`). Concurrent tool calls on the cached session MUST be safe.

## ADDED Requirements

### Requirement: Provider Config Resolution

The proxy MUST resolve the MCP provider configuration by provider ID at boot time. An unknown provider ID MUST surface `ErrUnknownProvider` to the caller without spawning any subprocess.

#### Scenario: Known provider

- GIVEN a proxy constructed with a valid provider config map
- WHEN a tool call arrives for a registered provider ID
- THEN the proxy resolves the provider's command, args, and `startup_timeout_s`

#### Scenario: Unknown provider

- GIVEN a proxy constructed with a valid provider config map
- WHEN a tool call arrives for an unregistered provider ID
- THEN the proxy returns `ErrUnknownProvider`
- AND no subprocess is spawned

### Requirement: Lazy-Spawn Persistent Lifecycle

The proxy MUST spawn the MCP client adapter on the FIRST tool call for a given provider. Subsequent tool calls for the same provider MUST reuse the cached `ClientSession`. The proxy MUST NOT spawn more than one subprocess per provider at any time.

#### Scenario: First call spawns subprocess

- GIVEN no active session for a provider
- WHEN the first tool call for that provider arrives
- THEN the proxy spawns the adapter, connects, and dispatches the call

#### Scenario: Subsequent calls reuse session

- GIVEN an active session for a provider
- WHEN a second tool call for the same provider arrives
- THEN the proxy dispatches through the existing session without spawning again

#### Scenario: Concurrent calls on same session

- GIVEN an active session for a provider
- WHEN multiple tool calls arrive concurrently for the same provider
- THEN all calls complete without data races or panics
- AND each call receives its own independent result

#### Scenario: Concurrent first calls spawn exactly once

- GIVEN no active session for a provider
- WHEN multiple tool calls arrive concurrently as the first calls
- THEN exactly one subprocess is spawned
- AND all calls are served by that single session

### Requirement: Session Reap on Application Stop

The proxy MUST close the `ClientSession` and terminate the subprocess when the application stops (`App.Stop` / proxy `Close`). After `Close`, no active session MUST remain in the cache.

#### Scenario: App stop terminates subprocess

- GIVEN an active session for a provider
- WHEN the application stops (proxy `Close` called)
- THEN the session's `Close` is called, the subprocess exits
- AND the session is removed from the cache

#### Scenario: Close guarded on context cancel — no leak

- GIVEN an active session for a provider
- WHEN the parent context is cancelled and then `Close` is called
- THEN the subprocess is still terminated
- AND no goroutines started by the proxy remain running (verified by goroutine-leak check)
- AND no OS process remains orphaned

#### Scenario: Close with no session ever spawned

- GIVEN the proxy was constructed but no tool call was ever made
- WHEN `Close` is called
- THEN no error is returned and no subprocess interaction occurs

### Requirement: Stale-data freshness via sidecar hot-reload

The long-lived sidecar MUST NOT require respawning to pick up `graph.json` changes — `graphify serve` hot-reloads on mtime change. The proxy performs no freshness management of its own.

#### Scenario: graph.json updated mid-lifetime

- GIVEN an active session whose graph.json is regenerated by INIT (`graphify update`)
- WHEN a tool call arrives after the file changed
- THEN the result reflects the updated graph (served by graphify's own mtime hot-reload)
- AND the proxy did not respawn the subprocess
