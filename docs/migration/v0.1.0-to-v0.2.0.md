```
spec id   : sophia-migration-v0.1.0-to-v0.2.0
authority : docs/specs/sophia-wire-v1.md + ADR-0003 + M10 plan + Phase 5 contract gate
status    : authoritative migration guide for the v0.1.0 â†’ v0.2.0 cut-over
applies   : sophia-cli AND sophia-orchestator (mirrored verbatim in both repos)
phase     : M10 Phase 6 deliverable (D-M10-16 release blocker #4)
```

> **TL;DR.** v0.2.0 is a **coordinated cut-over**. Upgrade both
> `sophia-cli` and `sophia-orchestator` at the same time. There is **no
> partial-upgrade path**. Plan a maintenance window, snapshot your
> orchestrator state, follow the checklist below, and validate with
> `sophia doctor` before resuming workloads.

---

## 1. Why this is a coordinated cut-over

`sophia-cli v0.1.0` and `sophia-orchestator v0.1.x` shipped before the
canonical wire spec existed. They evolved their HTTP and SSE surfaces
in parallel without cross-repo integration tests. v0.1.0 was never
end-to-end functional against the real orchestrator service â€” see the
`Known limitations` section of the v0.1.0 sophia-cli CHANGELOG for the
catalog of mismatches.

v0.2.0 is the first release where both repos implement a single
canonical specification: `docs/specs/sophia-wire-v1.md`, version `v1`.
The two repos carry byte-identical mirrors of the spec, gated by a
SHA256 cross-repo invariant in CI.

Because every wire surface changed simultaneously (paths, auth, SSE
model, error envelope), mixing v0.1.x with v0.2.0 on either side fails
on the FIRST request:

| Mixed pair | First-request failure |
|---|---|
| cli v0.2.0 â†’ orch v0.1.x | `GET /api/v1/health` â†’ 404 (orch only has `/healthz`); cli also sends `X-Sophia-API-Key` which the v0.1.x orch ignores; SSE subscribe to `/api/v1/phases/{id}/events` â†’ 404. |
| cli v0.1.0 â†’ orch v0.2.0 | `GET /api/v1/healthz` â†’ 404; orch demands `X-Sophia-API-Key` which the v0.1.0 cli does not send â†’ 401; SSE subscribe to `/api/v1/changes/{id}/events` â†’ 404. |

There is no "compat shim" on either side. Both repos rejected dual-API
mode in M10 Phase 1 (D-M10-06, D-M10-12) so the wire surface stays
minimal and the contract test gate stays meaningful.

## 3. What changed (wire-protocol summary)

The spec is the authoritative source; the table below is a directional
summary for operators.

### 3.1 Endpoints

| Surface | v0.1.x | v0.2.0 |
|---|---|---|
| Health | `GET /api/v1/healthz` | **`GET /api/v1/health`** (no `/healthz` alias on the orchestrator) |
| Approve a phase | `POST /api/v1/changes/{cid}/phases/{pid}/approve` | **`POST /api/v1/phases/{phase_id}/approve`** (phase-scoped, sophia-wire-v1 Â§4.3 / D-M10-13 Form A) |
| Reject a phase | `POST /api/v1/changes/{cid}/phases/{pid}/reject` | **`POST /api/v1/phases/{phase_id}/reject`** |
| Per-change SSE feed | `GET /api/v1/changes/{id}/events` | **gone**; replaced by per-phase streams (see SSE) |
| Per-phase SSE feed | n/a | **`GET /api/v1/phases/{phase_id}/events`** |

The Required endpoint surface for v0.2.0 is exactly:

```
GET  /api/v1/health
POST /api/v1/changes
GET  /api/v1/changes
GET  /api/v1/changes/{change_id}
POST /api/v1/changes/{change_id}/abort
GET  /api/v1/phases/{phase_id}
POST /api/v1/phases/{phase_id}/approve
POST /api/v1/phases/{phase_id}/reject
GET  /api/v1/phases/{phase_id}/events
```

### 3.2 Authentication

| Mode | v0.1.x | v0.2.0 |
|---|---|---|
| Header name | (none / inconsistent) | **`X-Sophia-API-Key`** (canonical; the legacy `X-API-Key` is accepted by the server but clients SHOULD use the canonical header). |
| Local loopback anonymous | informal | **explicit:** orchestrator MUST be bound to a loopback interface (`localhost`, `127.0.0.0/8`, `::1`) **AND** `AllowAnonLocalhost=true` for anonymous access to be permitted. The flag is **silently downgraded** to `false` when the listener is bound to a non-loopback address; a warning is logged. (D-M10-02.) |
| Remote orchestrator | informal | **MUST** present a valid API key via `X-Sophia-API-Key`. The cli refuses to make a request without one, exiting `3` with `auth required for remote orchestrator (set SOPHIA_API_KEY or --api-key)` BEFORE the HTTP call. |

The cli resolves the key from (in order): `--api-key` flag,
`SOPHIA_API_KEY` env var, then nothing. The key is never logged or
printed; commands that don't talk to the orchestrator (`version`,
`doctor`, `init`, `start`, `stop`, `help`) skip the auth gate.

### 3.3 SSE event stream

| Aspect | v0.1.x | v0.2.0 |
|---|---|---|
| URL shape | per-Change | **per-Phase** (sophia-wire-v1 Â§4.3 / D-M10-05) |
| Event id | RFC3339Nano timestamp (could collide) | **ULID per event** (`Last-Event-ID` resume safe) |
| Approve/reject events | `phase.approved` + `phase.rejected` (split) | unified **`approval.resolved`** with `decision: "approved" \| "rejected"` |
| Agent dispatch | `agent.spawned` (legacy) | canonical **`agent.dispatched`** (cli accepts both for backward-compat with prior fixtures) |
| Terminal phase, no further events | undefined | **`410 phase_terminal_no_events`** â€” clients fall back to `GET /api/v1/phases/{id}` for state |
| Optional `apply.*` diagnostic events | n/a | tolerated; clients MUST forward unknown events unchanged (sophia-wire-v1 Â§10) |
| `open` event on connect | n/a | sent first for fast reconnect detection |

The cli's `sophia run` and `sophia attach` use a **per-phase
multiplexer**: subscribe to the current phase, drain events, snapshot
the change, re-subscribe when `current_phase_id` changes, finish when
the change is terminal.

### 3.4 Error envelope

All non-2xx responses now carry the canonical envelope (sophia-wire-v1
Â§9.1):

```json
{
  "code": "<one of 13 stable codes>",
  "error": "<human-readable message>",
  "details": { "...optional structured fields..." }
}
```

The 13 stable codes (sophia-wire-v1 Â§9.2):
`unauthorized`, `validation_failed`, `approver_required`,
`limit_too_large`, `change_not_found`, `phase_not_found`,
`change_already_exists`, `change_already_terminal`,
`phase_not_resumable`, `phase_not_gated`, `gate_already_decided`,
`phase_terminal_no_events`, `internal_error`.

Clients SHOULD switch on `code` rather than the human-readable `error`
field, and SHOULD use a default branch for unknown codes (forward-compat).

### 3.5 New CLI commands

`sophia approve <phase-id>`, `sophia reject <phase-id>`, and
`sophia abort <change-id>` are new in v0.2.0. The browser-based `[O]`
keybinding from M7 is unchanged â€” both decision channels resolve the
same gate (D-M10-03). `gate_already_decided` and
`change_already_terminal` are surfaced as informational messages with
exit code 0 (idempotent UX).

## 4. Operator upgrade checklist

Run this checklist top-to-bottom in a maintenance window. Each step
should complete cleanly before proceeding.

### Pre-upgrade

- [ ] **Snapshot orchestrator state.** Dump `sophia-orchestator`'s
  Postgres database. v0.2.0 carries no destructive migrations beyond
  what the orchestrator's own `migrate-up` flow performs, but a
  rollback path requires a snapshot.
- [ ] **Decide your auth posture.**
  - **Local-only deployment** (loopback bind + single operator):
    you may keep `AllowAnonLocalhost=true` and skip API key
    provisioning. The orchestrator must bind to `localhost`,
    `127.0.0.0/8`, or `::1` for the flag to take effect; non-loopback
    binds are silently downgraded to "auth required".
  - **Remote / multi-user deployment**: provision an API key (random
    256-bit hex is fine). The key is stored in the orchestrator's
    config (`HTTP.APIKey`) and distributed to clients via
    `SOPHIA_API_KEY` env var or `--api-key` CLI flag.
- [ ] **Confirm the spec checksum** matches across both repos:
  ```bash
  shasum -a 256 docs/specs/sophia-wire-v1.md
  ```
  Both repos at the v0.2.0 SHAs MUST print the same digest.

### Upgrade

- [ ] **Step 1: stop traffic.** Drain in-flight Changes, or accept
  that any non-terminal Change at cut-over time will need to be
  manually re-driven after the upgrade. The orchestrator's
  `interrupted` phase status (sophia-wire-v1 Â§6.2) covers this.
- [ ] **Step 2: upgrade the orchestrator first.** Stop the v0.1.x
  binary, deploy v0.2.0, run migrations
  (`make migrate-up` in the orchestrator repo or your equivalent
  deploy hook), start the v0.2.0 binary. Confirm it answers
  `GET /api/v1/health` (NOT `/healthz`).
  ```bash
  curl -sf http://<orch-host>/api/v1/health
  # â†’ {"status":"ok","version":"v1",...}
  ```
- [ ] **Step 3: configure the API key on each cli host** (remote
  deployments only):
  ```bash
  export SOPHIA_API_KEY=<your-key>
  ```
  Or pass `--api-key=<your-key>` per command. Do **not** check the
  key into shell rc files that are world-readable; `0600`-permissioned
  per-user secrets stores are recommended.
- [ ] **Step 4: upgrade the cli** on each operator host:
  ```bash
  go install github.com/RVRTelecomunicaciones/sophia/cmd/sophia@v0.2.0
  # or: download the v0.2.0 binary from the release page.
  sophia version
  # â†’ expect v0.2.0
  ```

### Post-upgrade validation

- [ ] **Step 5: `sophia doctor`** â€” confirms the cli reaches the
  upgraded orchestrator at the new health path.
  ```bash
  sophia doctor
  # â†’ orchestrator: 200 OK
  # â†’ SSE handshake: deferred to first run/attach (info-level, expected)
  ```
- [ ] **Step 6: `sophia changes`** â€” confirms List works against the
  new envelope shape and the new auth header.
  ```bash
  sophia changes --limit 10
  ```
- [ ] **Step 7: smoke `run â†’ detach â†’ attach`.** Validates the new
  per-phase SSE multiplexer end-to-end.
  ```bash
  sophia run "smoke validation post-upgrade"
  # In the TUI: press [Q] to detach (preserves the change_id).
  sophia attach <change-id>
  # Should reattach and show the timeline + ApplyBoard.
  ```
- [ ] **Step 8: re-enable traffic.** Resume normal workload routing.

### Failure handling during upgrade

If any step fails:

1. Capture the failing command's output (`sophia <cmd> --json` is
   machine-readable).
2. If the failure is `auth: auth required for remote orchestrator â€¦`,
   set `SOPHIA_API_KEY` (or `--api-key`) and retry.
3. If the failure is `orchestrator: HTTP 404` or
   `connection refused`, the orchestrator likely did not finish
   restarting. Wait, then retry `sophia doctor`.
4. If the failure is anything else, see the **Rollback** section.

## 5. Rollback

Rollback is **coordinated**, just like the upgrade. There is no
partial-rollback path.

### When to roll back

Roll back when:

- `sophia doctor` keeps failing with HTTP 4xx/5xx on the orchestrator
  check after a 5-minute soak.
- `sophia run` / `attach` smoke fails with errors that do **not** map
  to the documented stable codes (sophia-wire-v1 Â§9.2).
- The orchestrator log reports persistent panics or migration
  failures.

### Rollback procedure

- [ ] **Step R1: stop traffic.** Same as upgrade Step 1.
- [ ] **Step R2: roll back the orchestrator FIRST.** Stop the v0.2.0
  binary, restore the pre-upgrade Postgres snapshot, redeploy the
  v0.1.x binary.
- [ ] **Step R3: roll back the cli on each operator host.** Reinstall
  v0.1.0 on every host where v0.2.0 was deployed:
  ```bash
  go install github.com/RVRTelecomunicaciones/sophia/cmd/sophia@v0.1.0
  ```
- [ ] **Step R4: clear the API key environment** (if you set it during
  upgrade and the v0.1.x deployment doesn't expect one):
  ```bash
  unset SOPHIA_API_KEY
  ```
- [ ] **Step R5: revalidate.** Run `sophia doctor` and confirm the
  orchestrator check passes against the rolled-back endpoints.

### Mixed-version warnings

While rolling back, do **not**:

- Run cli **v0.2.0** against orchestrator **v0.1.x** at any point. The
  cli will fail on the very first request (`/api/v1/health` is a 404
  on v0.1.x).
- Run cli **v0.1.0** against orchestrator **v0.2.0** at any point. The
  cli will fail on the very first request (it sends
  `GET /api/v1/healthz` which v0.2.0 does not serve, and it doesn't
  send `X-Sophia-API-Key` which v0.2.0 requires for remote orchestrators).

If a partial deployment exists transiently (e.g. some operator hosts
still on v0.1.0 cli during rollback), the affected hosts will produce
errors but will not corrupt orchestrator state â€” every cli command is
either read-only or idempotent on the orchestrator side.
